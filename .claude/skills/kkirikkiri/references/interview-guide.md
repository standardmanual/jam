# 끼리끼리 인터뷰 가이드

> 유저의 모호한 요청을 2-3개 질문으로 구체화하는 방법론.
> 핵심: 유저가 아는 것(목적, 대상)은 질문으로, 모르는 것(팀 구성, 모델)은 AI가 결정.

---

## 인터뷰 설계 규칙

| 규칙 | 설명 | 예시 |
|------|------|------|
| 설명 필수 | 모든 옵션에 "뭔지 + 장점 + 단점" | "종합 리포트 — 깊이 있는 분석. 시간 좀 걸림." |
| 추천 표시 | 가장 적합한 옵션에 (추천) + 첫 번째 배치 | "종합 리포트 (추천)" |
| 비유 사용 | 기술 용어 대신 일상 비유 | "Opus" → 사용하지 않음 (유저에게 모델명 노출 X) |
| 복잡도 힌트 | 설명에 기대치 표시 | "시간 좀 걸림", "빠르게" |
| 최소 질문 | 프리셋별 2-3개. 절대 4개 초과 금지 | |
| 열린 질문 최소 | 첫 질문(뭘 하고 싶어?)만 열린 질문 | |
| 잘 모르겠어요 | 모든 질문에 "추천대로" 옵션 존재 | |
| 모델명 비노출 | 유저에게 Opus/Sonnet/Haiku 이름 노출 금지 | 내부적으로만 사용 |

## 인터뷰 흐름

### Phase 1: 의도 파악 (자동)

유저의 자연어 입력에서 키워드를 매칭하여 프리셋 결정.

```
"경쟁사 비교 리서치 해줘" → keywords: [리서치, 비교] → research 프리셋
"이 기능 만들어줘" → keywords: [만들어줘] → development 프리셋
"코드 구조 파악해줘" → keywords: [파악해줘, 구조] → analysis 프리셋
"README 써줘" → keywords: [써줘, README] → content 프리셋
"도움이 필요해요" → 매칭 실패 → generic (범용 인터뷰)
```

### Phase 2: 프리셋별 인터뷰 (AskUserQuestion)

매칭된 프리셋의 인터뷰 질문을 순서대로 진행.
(질문 내용은 presets.md 참조)

AskUserQuestion 호출 시 규칙:
- questions 배열에 최대 3개 질문 (4개 초과 금지)
- 각 question에 header (12자 이내), options (2-4개)
- 각 option에 label, description 필수
- 첫 번째 option이 추천 (label에 "(추천)" 포함)
- multiSelect는 명시된 경우만 true

### Phase 3: 환경 스캔 (자동)

인터뷰와 병렬로 환경 스캔 실행:

```
1. 외부 AI CLI 확인
   - Bash: command -v codex → 코드 리뷰/분석 역할에 활용
   - Bash: command -v gemini → 디자인/대규모 분석에 활용

2. MCP 서버 확인
   - Perplexity → 리서치 팀원에게 웹 검색 도구 배정
   - Context7 → 기술 문서 조사 도구 배정

3. 기존 에이전트/플러그인 파일
   - Glob: .claude/agents/*.md
   - 프로젝트 내 관련 파일 탐색

4. CLI 도구
   - command -v gh → PR/이슈 관리
   - command -v npm, command -v bun → 빌드/테스트
```

### Phase 4: 팀 구성 제안 (유저 확인)

인터뷰 + 환경스캔 결과를 종합해 팀 구성을 제안하고 유저 확인.

제안 형식:
```
"이렇게 팀을 구성할게요:

├── 팀장 — 전체 계획 수립 + 결과 통합
├── 리서처 1 — 웹 리서치
├── 리서처 2 — 문서 분석
└── (Codex) — 기술 코드 분석 (백그라운드)

이대로 진행할까요?"
```

유저에게 모델명(Opus/Sonnet)은 보여주지 않음.
역할과 하는 일만 보여줌.

## 나쁜 질문 vs 좋은 질문

❌ 나쁜 질문:
"팀 규모를 선택하세요: [3명] [5명] [7명]"
→ 몇 명이 적당한지 감이 없음

✅ 좋은 질문:
"리서치 깊이는?"
- "깊고 포괄적으로 (추천)" — 여러 소스 교차 검증. 시간 더 걸림.
- "빠르게 핵심만" — 30분 내외. 깊이 제한.
- "잘 모르겠어요" — 추천대로 갈게요.
→ 유저가 원하는 것(깊이)을 물어보면 팀 규모는 AI가 결정

## 바이브코더 대응 전략

| 유저 상태 | 대응 |
|----------|------|
| 뭘 원하는지 모호 | 범용 인터뷰로 전환, "이 중에 가까운 게 있나요?" |
| 선택지를 이해 못함 | 비유로 재설명, 추천 옵션 강조 |
| 전부 "잘 모르겠어요" | 모든 추천대로 진행 (기본값이 항상 합리적) |
| 너무 많은 걸 원함 | "한 번에 다 하면 복잡해요. 이것부터 시작할까요?" |
| 기존 경험 언급 | "어떤 점이 불편했어요?" → 팀 구성에 반영 |

## AskUserQuestion 호출 패턴

모든 인터뷰는 Claude Code의 AskUserQuestion 도구를 통해 진행.

호출 규칙:
- 한 번에 최대 3개 질문 (questions 배열)
- 각 질문의 options는 2-4개
- multiSelect는 명시적으로 필요한 경우만 true
- header는 12자 이내 한글
- "Other" 옵션은 AskUserQuestion이 자동 제공 (직접 추가 금지)
