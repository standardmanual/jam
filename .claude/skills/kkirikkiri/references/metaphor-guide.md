# 비유 가이드 — 기술 용어 → 일상 표현 변환표

> 끼리끼리는 비개발자도 사용합니다.
> 기술 용어를 그대로 노출하면 안 됩니다. 이 가이드의 표현을 사용하세요.

---

## 모델명

| 내부 용어 | 사용자에게 보여줄 표현 | 언제 쓰는지 |
|----------|---------------------|------------|
| Opus | "가장 똑똑한 AI" | 팀장, 핵심 판단이 필요한 역할 |
| Sonnet | "전문 AI" 또는 "균형잡힌 AI" | 실행력이 필요한 팀원 |
| Haiku | (사용 금지) | 어떤 역할에도 배정하지 않음 |
| Codex CLI | "코드 전문 AI" | 코드 분석/리뷰 특화 |
| Gemini CLI | "대규모 분석 AI" | 큰 파일/많은 문서 처리 |

### 예시

```
❌ "Opus 모델을 팀장으로 배정합니다"
✅ "가장 똑똑한 AI가 팀장을 맡아요"

❌ "Sonnet 2명을 리서처로 스폰합니다"
✅ "전문 AI 2명이 리서치를 담당해요"

❌ "Codex CLI로 코드 리뷰를 수행합니다"
✅ "코드 전문 AI가 코드를 검토해요"
```

## 시스템 용어

| 내부 용어 | 사용자에게 보여줄 표현 | 설명 |
|----------|---------------------|------|
| Agent Teams | "AI 팀" | 여러 AI가 협력하는 그룹 |
| TeamCreate | (언급하지 않음) | 내부 API — 사용자가 알 필요 없음 |
| Task / spawn | "팀원을 불러오다" | 에이전트 생성 |
| SendMessage | (언급하지 않음) | 내부 통신 |
| shutdown_request | "작업 종료" | 팀 해산 |
| MCP | "연동 도구" | 외부 서비스 연결 |
| sub-agent / 심부름꾼 | "도우미" 또는 "보조 AI" | 팀원이 부르는 임시 AI |
| shared memory | "작업 기록" | 팀이 공유하는 파일 |
| TEAM_PLAN.md | "작업 계획 파일" | 내부 파일명은 언급하지 않음 |
| TEAM_PROGRESS.md | "진행 상황 파일" | 내부 파일명은 언급하지 않음 |
| TEAM_FINDINGS.md | "발견 사항 파일" | 내부 파일명은 언급하지 않음 |
| DEAD_ENDS | "실패한 시도 기록" | 같은 실수 반복 방지용 |
| Ralph Pattern | "자동 재시도" | 품질 부족 시 반복 |
| TeammateIdle | (언급하지 않음) | 내부 상태 |
| preset | "팀 유형" 또는 "팀 템플릿" | 사전 정의된 팀 구성 |

### 예시

```
❌ "Agent Teams를 TeamCreate로 생성하고 Task로 에이전트를 스폰합니다"
✅ "AI 팀을 만들고 팀원들을 불러올게요"

❌ "MCP를 통해 Perplexity API에 연결합니다"
✅ "웹 검색 도구를 활용할게요"

❌ "TEAM_FINDINGS.md의 DEAD_ENDS 섹션에 기록합니다"
✅ "실패한 시도를 기록해서 같은 실수를 반복하지 않아요"

❌ "Ralph Pattern으로 검증 루프를 돌립니다"
✅ "결과가 부족하면 자동으로 보강해요"
```

## 품질/검증 표현

| 내부 개념 | 사용자에게 보여줄 표현 |
|----------|---------------------|
| 품질 판정 PASS | "잘 되었어요" |
| 품질 판정 FAIL | "부족한 부분이 있어요" |
| 바이어스 누적 | (언급하지 않음 — 내부 판단 기준) |
| 방식 A (팀 유지) | "기존 팀이 보강해요" |
| 방식 B (재구성) | "새 팀으로 다시 시도해요" |
| 방식 C (부분 교체) | "일부 팀원을 교체해요" |
| kill criteria | (언급하지 않음 — 자동 처리) |
| blind review | (언급하지 않음 — 내부 검증 방식) |

## 일반 원칙

1. **기술 용어는 절대 그대로 노출하지 않는다** — 이 표에서 대응하는 표현을 찾아 사용
2. **표에 없는 기술 용어가 나오면** — "AI가 [행위]를 합니다" 형태로 변환
3. **영어 용어도 피한다** — spawn → 불러오기, shutdown → 종료, verify → 확인
4. **파일명/경로는 언급하지 않는다** — 내부 구현 세부사항
5. **에러 메시지는 번역한다** — 기술적 에러 → "문제가 생겨서 다른 방법으로 처리했어요"
