### 기본 구성 (3명)

| Developer 2 | Opus | 보조 기능 구현 (UI, 유틸리티, 설정) | 메인 로직 변경, DB 스키마 변경 |

- 없으면: Lead(Opus)가 리뷰 직접 수행

### 인터뷰 질문 (최대 3개)

Q1. "뭘 만들고 싶으세요?" (header: "목표")
    → 열린 질문 (유저 자유 입력)

Q2. "기존 코드가 있나요?" (header: "기존 코드", multiSelect: false)
    options:
    - "새로 시작 (추천)" — 처음부터 만들기. 기존 코드에 영향 없음. 가장 깔끔.
    - "기존 코드에 추가" — 현재 프로젝트에 기능 추가. 기존 패턴/구조를 따라야 함.
    - "기존 코드 수정/리팩토링" — 있는 코드를 고치기. 기존 테스트가 깨지지 않게 주의.
    - "잘 모르겠어요" — 프로젝트를 스캔해서 판단할게요.

Q3. "테스트도 필요해요?" (header: "테스트", multiSelect: false)
    options:
    - "네, 테스트도 같이 (추천)" — 테스터 역할 추가. 품질 높지만 시간 더 걸림.
    - "아니요, 구현만" — 기본 3명으로 빠르게. 테스트는 나중에.
    - "잘 모르겠어요" — 추천대로 테스트 포함할게요.

### 동적 조정 규칙
- Q3에서 "테스트도 같이" → 확장 구성 (Tester 추가)
- Q2에서 "기존 코드에 추가" → 환경스캔으로 기존 패턴/스택 파악 후 팀에 전달
- 환경스캔에서 gh CLI 발견 → PR 생성 기능 활용 제안

## 공통 규칙

### 팀장(Lead) R&R — 모든 프리셋 공통
- 팀장은 무조건 Opus
- 팀장은 코드를 짜지 않는다 (계획/배분/검증/통합만)
- 팀장은 PM이다 — 직접 구현이 아닌 명확한 지시
- 팀장이 직접 작업하면 R&R 위반

### 모델 배정 기본 원칙
- Lead: 항상 Opus
- 핵심 작업자: Opus 우선
- 보조/단순 작업: Sonnet (최소한으로)
- Haiku: 사용하지 않음
- 외부(Codex/Gemini): CLI 백그라운드 실행, 없으면 Claude 폴백

### 동적 조정 공통 원칙
- 프리셋은 출발점, 고정값 아님
- 인터뷰 답변으로 역할 수/모델 변경
- 환경스캔으로 도구/리소스 반영
- 같은 프리셋이라도 매번 다른 팀 구성 가능
