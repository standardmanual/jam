## Phase 8: 아이템북 완성 루프 (예상 2주)

### 목표
아이템북 레시피를 모두 채우면 보상 배지가 자동 발급되는 완성 루프 구현.
현재 아이템북 UI는 있지만 완성 조건 충족 시 아무 일도 일어나지 않는 상태.

### 기능
- **아이템북 완성 감지**: 인벤토리에 required_item_badge_ids가 모두 갖춰지면 자동 감지
- **완성 시 보상 발급**: reward_badge가 있으면 user_activity_badges에 자동 INSERT
- **완성 알림**: "아이템북 완성! 보상 배지를 획득했어요" 화면 내 알림
- **완성된 아이템북 UI**: 아이템북 목록에서 완성 여부 표시 (뱃지/그레이스케일)
- **Strava 동기화 시 트리거**: 배지 발급 직후 아이템북 완성 체크 파이프라인 추가

### 비즈니스 가치
아이템북은 JAM!의 핵심 콘텐츠 소비 루프. 완성 → 보상 → 새 목표 생성의 사이클이 없으면 인벤토리가 쌓여도 의미 없음.

### 전제 조건
Phase 3 (인벤토리), Phase 6 (아이템 드랍)

## Phase 9: 아이템 만료 + JAM 포인트 (예상 3주)

### 목표
인벤토리 아이템의 30일 만료 정책 실행 + 만료 시 JAM 포인트로 자동 환전.

### 기능
- **만료 Cron Job**: 매일 자정 `expires_at < NOW()` 아이템 자동 처리
- **JAM 포인트 테이블**: `user_points` (user_id, balance, updated_at)
- **포인트 내역 테이블**: `point_transactions` (user_id, amount, reason, created_at)
- **만료 환전**: 아이템 rarity별 포인트 환산 (common: 10P, rare: 30P, legendary: 100P, mythic: 500P)
- **포인트 잔액 표시**: 프로필 화면에 JAM 포인트 잔액 표시
- **포인트 사용처** (이후 Phase 연계):
  - 특별 드랍 이벤트 참가권
  - 인벤토리 슬롯 확장 (기본 50 → 최대 100)

### 비즈니스 가치
아이템 만료가 "손해"가 아닌 "환전"이 되면 유저 이탈 방지. 포인트 시스템은 이후 인앱 경제의 기반.

### 전제 조건
Phase 3 (인벤토리 + 아이템 만료 필드)

## Phase 10: 푸시 알림 / 인게이지먼트 (예상 3주)

### 목표
핵심 이벤트 발생 시 유저에게 알림을 보내 재방문율 향상.

### 기능
- **Web Push 구독**: Service Worker + Push API (PWA 필수 선행)
- **알림 종류**:
  - 배지 발급: "오늘 라이딩으로 새 배지를 획득했어요!"
  - 드랍 픽업됨: "당신이 드랍한 [배지명]을 누군가 픽업했어요"
  - 아이템북 완성: "[아이템북 이름] 완성! 보상 배지를 확인하세요"
  - 만료 임박: "아이템 3개가 7일 후 만료됩니다"
  - Strava 동기화 완료: "오늘의 활동이 동기화됐어요. 새 배지 1개!"
- **알림 설정**: 유저별 알림 ON/OFF 선택 (프로필 화면)
- **어드민**: 전체 공지 푸시 발송 기능

### 비즈니스 가치
JAM!은 "저녁에 앱을 열면 배지가 쌓이는" 경험이 핵심. 알림 없이는 재방문 동기가 약함. DAU 직결 기능.

### 전제 조건
PWA 설정 (manifest.json + Service Worker), HTTPS 필수

## Phase 11: 커뮤니티 피드 + 소셜 공유 (예상 4주)

### 목표
다른 유저의 배지 획득, 드랍/픽업 활동이 피드에 노출되어 경쟁/참여 동기 유발.

### 기능
- **팔로우 시스템**: `user_follows` 테이블 (follower_id, following_id)
- **활동 피드**: 팔로우한 유저의 배지 획득, 드랍 완료, 아이템북 완성 노출
- **전체 피드**: 팔로우 없어도 볼 수 있는 지역 기반 전체 피드
- **인스타 공유 카드 고도화**:
  - 배지 + 일련번호 + "JAM! 앱에서 확인" QR코드 합성
  - 공유 후 링크로 해당 배지 상세 직접 진입
- **랭킹**: 이번 달 배지 획득 수 기준 지역별 랭킹 (Top 10)
- **유저 프로필 공개**: `/u/[userId]` — 공개 배지 컬렉션 페이지

### 비즈니스 가치
SNS 확산이 JAM! 신규 유입의 핵심 채널. 공유 카드 → 링크 → 가입의 바이럴 루프.

### 전제 조건
Phase 1 (배지 발급), Phase 10 (알림 — 팔로우 알림 연계)

## Phase 12: 실물 패치 D2C 연동 (예상 2주)

### 목표
배지 달성 시에만 구매 가능한 실물 자수 패치 구매 플로우 연결.
"돈이 있어도 아무나 살 수 없음"이 JAM! 핵심 차별점.

### 기능
- **패치 구매 가능 배지 표시**: 배지 상세에 "실물 패치 구매 가능" 뱃지 (patch_available = true)
- **외부 D2C 스토어 연결**: 배지별 패치 상품 URL로 아웃링크
  - 배지 달성 여부 검증 후 버튼 활성화 (미달성 시 잠금 + 조건 안내)
- **구매 인증 코드**: 달성 유저에게 1회용 할인/인증 코드 발급 (어드민 생성)
- **패치 컬렉션 화면**: 구매 가능/구매 완료 패치 모아보기
- **어드민**: 패치 상품 URL 관리, 인증 코드 발급

### 비즈니스 가치
JAM!의 첫 번째 수익 모델. DAU를 매출로 전환하는 핵심 퍼널.

### 전제 조건
Phase 1 (배지 발급), D2C 쇼핑몰 외부 구축 필요

## Phase 13: P2P 플리마켓 (예상 5주)

### 목표
유저 간 아이템 배지 1:1 교환 거래. 기존 Phase 3에서 "준비 중"으로 잠근 플리마켓 실제 오픈.

### 기능
- **거래 요청**: `trades` 테이블 활성화 (이미 스키마 존재)
  - 내 아이템 제시 + 상대 아이템 요청 → 상대에게 거래 요청 전송
- **거래 수락/거절**: 요청받은 유저가 수락/거절
- **원자 교환**: 트랜잭션으로 두 아이템 동시 교환 (중간 실패 없음)
- **거래 알림**: "거래 요청이 왔어요" / "거래가 수락됐어요" 알림
- **거래 이력**: 내가 보낸/받은 거래 내역 조회
- **사기 방지**:
  - 거래 확인 후 24시간 내 취소 불가
  - 동일 아이템 중복 요청 차단
  - rate_limit: 1일 최대 10건

### 비즈니스 가치
커뮤니티 활성화 + 희귀 아이템 수요/공급 생성 → 플리마켓 자체가 콘텐츠.

### 전제 조건
Phase 3 (인벤토리), Phase 10 (알림), Phase 11 (유저 간 관계)

## Phase 14: 지도 고도화 + POI 확장 (예상 3주)

### 목표
드랍/픽업 지도 UX 개선 + T3 POI (체육관, 공원 입구 등) 확장.

### 기능
- **마커 클러스터링**: 줌아웃 시 밀집 마커 그룹화
- **드랍 히트맵**: 내 위치 기준 드랍 밀도 시각화
- **드랍 이력**: 내가 드랍한 아이템 현황 (픽업됐는지 확인) + 내가 픽업한 이력
- **T3 POI 추가**: 헬스장, 공원 입구, 학교 등 생활 POI
  - OSM 태그: `leisure=fitness_centre`, `leisure=park`, `amenity=school`
- **POI 거리 표시**: 지도 하단에 "가장 가까운 POI: CU 서초점 32m" 상시 표시
- **오프라인 대응**: 위치 없을 때 주변 드랍 목록 텍스트 뷰로 폴백

### 전제 조건
Phase 7 (드랍/픽업 + OSM T2)

## Phase 15: 아이템 조합/믹싱 — 비밀 조합법 시스템 (예상 3주)

### 목표
인벤토리에 쌓이는 잉여 아이템을 소각하고, 조합 공식 바이럴로 유저 간 자발적 커뮤니티를 형성한다.

> 서비스 플랜 4.5: "가방 공간을 차지하는 2~3개의 잉여 아이템을 가상의 조합기에 넣고 합성하면 원본 아이템들은 영구 소각되고, 일정 확률로 새로운 희귀 아이템이 탄생한다."

### 기능

**데이터 모델 추가**
- `combination_recipes` 테이블:
  - `id`, `ingredient_badge_ids` (UUID 배열, 2~3개), `result_badge_id`, `success_rate` (0.0~1.0), `is_public` (공개 여부), `created_at`
  - `is_public = false` → 힌트만 노출, 조합법 숨김 (바이럴 유도)

**조합 API** `POST /api/combine`
- 바디: `{ item_ids: [uuid, uuid] }` — 인벤토리 아이템 ID 2~3개
- 서버에서 조합 레시피 매칭 → 성공 시 원본 소각 + 결과 아이템 인벤토리에 추가
- 실패 시 원본 아이템 그대로 반환 (소각 없음) 또는 관리자 설정으로 소각 가능

**힌트 시스템**
- 조합 화면에서 `is_public = false` 레시피는 재료 실루엣 + 힌트 텍스트만 노출
  - 예: "🧤 ???? + ❄️ ???? → ????  힌트: 겨울 등반에 필요한 것들"
- `is_public = true` 레시피는 재료 공개 (B2B 브랜드 레시피 등)

**조합기 UI** (`/combine`)
- 인벤토리에서 아이템 2~3개 선택 → "합성하기" 버튼
- 합성 애니메이션 (결과 아이템 등장)
- 결과: 성공 → 새 아이템 획득 알림 / 실패 → "이 조합은 없어요" 메시지

**어드민**
- 조합 레시피 CRUD (`/admin/recipes`)
- 재료 배지 선택, 결과 배지 선택, 성공률 설정, 힌트 문구 입력, 공개 여부 토글

### 비즈니스 가치
- **인벤토리 디플레이션 엔진**: 잉여 아이템이 소각되어 아이템 희귀도 유지
- **자발적 바이럴**: 비밀 조합법 발견 시 유저들이 커뮤니티/단톡방에 자발 공유 → 자연 유입
- **B2B 레시피 상품화**: 브랜드가 자신의 아이템을 포함한 공개 레시피를 구매 → 추가 수익원

### 전제 조건
Phase 3 (인벤토리), Phase 9 (포인트, 소각 보상 연계 시)

## Phase 16: 다이나믹 미션 시스템 — 라이브 운영 엔진 (예상 3주)

### 목표
주기적으로 업데이트되는 단기 미션을 제공하여 매일/매주 재방문 동기를 부여한다.

> 서비스 플랜 4.7: "매일 반복되는 출퇴근 걷기나 동네 라이딩의 지루함을 타파하기 위해 단기 목표와 타겟을 제시하는 라이브 운영 시스템."

### 기능

**데이터 모델 추가**
- `missions` 테이블:
  - `id`, `title`, `description`, `mission_type` (`distance` / `poi_visit` / `activity_count` / `item_collect`), `condition_json` (기준값), `reward_type` (`badge` / `points` / `item_badge`), `reward_id` (해당 보상 ID), `reward_points`, `starts_at`, `ends_at`, `max_completions` (선착순 상한, null=무제한), `created_at`
- `user_mission_completions` 테이블:
  - `id`, `user_id`, `mission_id`, `completed_at`
  - UNIQUE (user_id, mission_id)

**미션 타입별 달성 조건 예시**
| 타입 | 예시 |
|------|------|
| `distance` | "이번 주 토요일 자정까지 라이딩 50km 완료" |
| `poi_visit` | "오늘 밤 21시~24시 한강 합수부 통과" |
| `item_collect` | "10월 한 달간 [떡볶이] 배지 획득" |
| `activity_count` | "이번 주 3회 이상 활동" |

**달성 감지 파이프라인**
- Strava 동기화 직후 (`badge-engine` 이후) → 활성 미션 순회 → 조건 체크 → 달성 시 `user_mission_completions` INSERT + 보상 지급

**미션 UI** (`/missions`)
- 진행 중 미션 목록 (남은 시간 카운트다운)
- 달성률 프로그레스 바
- 선착순 미션: "상위 1,000명에게만 지급 — 현재 237/1,000명 달성"
- 완료된 미션 이력

**어드민** (`/admin/missions`)
- 미션 생성/수정/삭제, 기간 설정, 달성 현황 모니터링

### 비즈니스 가치
- **DAU 견인**: 단기 미션 마감 시한이 재방문 트리거 역할. Phase 10 (푸시 알림)과 결합 시 효과 극대화
- **라이브 운영 레버**: B2B 브랜드 캠페인과 연계하여 "트렉 라이딩 챌린지" 같은 유료 미션 이벤트 판매 가능
- **선착순 희소성**: 상위 N명 제한으로 즉각적 행동 유발

### 전제 조건
Phase 1 (Strava 동기화), Phase 8 (아이템북 완성 루프 — 미션 보상 연계)

## Phase 17: 신화 아이템 떠돌이 속성 — Wandering Mythic (예상 2주)

### 목표
특정 신화 아이템이 고정된 인벤토리에 머물지 않고 72시간마다 새 POI로 이동하며, 유저 커뮤니티가 위치를 제보하고 즉흥 라이딩을 유도하게 만든다.

> 서비스 플랜 4.8: "특정 신화 아이템은 한 유저의 가방에 영원히 머물지 않고, 72시간이 지나면 인벤토리를 탈출하여 다른 POI로 도망가 버립니다. 유저들은 커뮤니티에서 실시간 위치를 제보하며 이를 포획하기 위해 즉흥적인 라이딩을 떠납니다."

### 기능

**데이터 모델 추가**
- `badges` 테이블에 `is_wandering BOOLEAN DEFAULT FALSE` 컬럼 추가
- `wandering_mythic_state` 테이블:
  - `id`, `badge_id` (UNIQUE), `current_poi_id`, `holder_user_id` (현재 보유 유저, null=POI에 드랍된 상태), `placed_at`, `expires_at` (placed_at + 72h), `times_caught` (누적 포획 횟수)

**떠돌이 사이클 로직** (Cron Job, 1시간마다 실행)
```
1. wandering_mythic_state에서 expires_at < NOW() 항목 조회
2. 보유 유저가 있으면 인벤토리에서 자동 소각 + 유저에게 "아이템이 탈출했어요!" 알림
3. 랜덤 활성 POI 선정 → current_poi_id 업데이트, holder_user_id = null, placed_at = NOW(), expires_at = NOW() + 72h
4. 지도에 해당 POI에 신화 드랍 마커 표시
```

**포획 플로우**
- POI 반경 50m 내에서 픽업 → 일반 픽업과 동일한 API 사용, 단 `is_wandering` 배지 처리 분기
- 픽업 성공 시: `holder_user_id` = 픽업 유저, `expires_at` = 픽업 시각 + 72h
- 포획 달성 → "전설의 ○○를 포획했습니다! 72시간 동안만 보유 가능" 특수 알림

**지도 UI 연동**
- 지도에 신화 아이템 위치 특수 마커 표시 (일반 드랍과 다른 색/아이콘)
- 마커 탭 시: "72시간 후 이동" 카운트다운 + 아이템 정보

**어드민**
- 떠돌이 신화 아이템 ON/OFF 토글
- 현재 위치/보유자 실시간 모니터링
- 강제 이동(관리자 재배치) 기능

### 비즈니스 가치
- **즉흥 라이딩 유발**: "지금 잡지 않으면 사라진다"는 긴급성이 즉각적 오프라인 행동 유발 (JAM!의 핵심 철학)
- **커뮤니티 이벤트 자동 생성**: 위치 제보 문화 → 자전거/러닝 커뮤니티 단톡방에 자발적 바이럴
- **한정판 굿즈 트리거**: 누적 포획 횟수가 10회 이상인 유저에게 신화 실물 패치 구매 자격 부여 가능

### 전제 조건
Phase 7 (드랍/픽업 시스템), Phase 10 (푸시 알림 — 탈출 알림 연계)

## Phase 18: 데이터 무결성 정책 — 운동 데이터 신뢰성 엔진 (예상 2주)

### 목표
이기종 기기/앱에서 동일 운동이 중복 인식되는 현상 방지, 차량 탑승 어뷰징 필터링, 드랍 아이템 자동 만료 정책을 확립하여 게임 경제 밸런스를 보호한다.

> 서비스 플랜 섹션 8: "미결정 및 향후 아젠다" 3개 항목 해소.

### 기능

**① 이기종 데이터 중복 방지**
- 문제: Strava + Apple HealthKit + Garmin이 동시에 켜진 경우 동일 운동이 3번 동기화됨
- 해결:
  - Strava Webhook 수신 시 `external_activities` 테이블에 `(user_id, strava_activity_id)` UNIQUE 제약으로 중복 INSERT 차단 (이미 일부 구현)
  - 동일 시간대(±5분 이내) + 동일 유저의 활동이 복수 입력될 경우 가장 최근 수신 우선, 나머지 무시
  - 어드민에서 중복 감지 로그 조회 가능

**② 차량 탑승 어뷰징 속도 필터**
- 문제: 자전거를 차량/버스에 싣고 이동 시 고속 라이딩으로 인식됨
- 해결:
  - Strava 활동의 `average_speed` > 임계값(기본 60km/h) 구간 데이터를 배지 발급 계산에서 제외
  - `abusing_policy` 테이블에 `vehicle_speed_filter_kmh INTEGER DEFAULT 60` 컬럼 추가 (Phase 7 어뷰징 정책 테이블 확장)
  - 단, 활동 자체를 무효화하지 않음 — 해당 구간만 거리/고도 계산에서 제외

**③ 유저 드랍 아이템 30일 자동 만료**
- 문제: POI에 드랍된 아이템이 수개월 방치 시 DB 비대화
- 해결:
  - `poi_drops` 테이블에 `expires_at TIMESTAMPTZ DEFAULT (NOW() + INTERVAL '30 days')` 컬럼 추가
  - 매일 자정 Cron Job: `expires_at < NOW() AND is_available = TRUE` 드랍 자동 소각
  - 소각 시 드랍 유저에게 소량 JAM 포인트 환전 (Phase 9 포인트 시스템 연계)
  - 어드민에서 만료 전 드랍 현황 조회 + 강제 만료/연장 가능

### 비즈니스 가치
- **경제 붕괴 방지**: 이기종 중복이 막히지 않으면 특정 유저가 같은 운동으로 배지를 3배 쌓는 부정행위로 이어짐
- **DB 비용 절감**: 30일 만료 정책으로 방치 드랍 데이터 누적 방지
- **어뷰징 정책 완결**: Phase 7 GPS 조작 감지에 이어 "속도 기반 차량 탑승 감지"로 어뷰징 방어 계층 추가

### 전제 조건
Phase 1 (Strava 동기화), Phase 7 (어뷰징 정책 테이블), Phase 9 (JAM 포인트 — 만료 환전 연계)
