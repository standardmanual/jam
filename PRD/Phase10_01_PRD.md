# JAM! Phase 10 PRD — 유저 검색

> 작성일: 2026-07-21  
> 범위: 홈 화면 유저 검색창, 검색 결과 페이지, 기존 유저 프로필 페이지(`/[username]`) 연결

---

## 1. 배경 및 목표

Phase 9에서 username 체계와 유저 프로필 페이지(`/[username]`)가 도입되었으나, 다른 유저를 **찾을 방법이 없다**. 프로필 URL을 직접 알아야만 접근 가능해 팔로우·소셜 기능의 발견성(discoverability)이 사실상 0이다.

Phase 10의 목표:
- **홈 화면에 유저 검색 진입점 추가** — 스트라바 동기화 영역 바로 아래에 검색창 + 버튼 배치.
- **아이디(username) 또는 이메일로 검색** — 부분 일치 검색 지원.
- **검색 결과 페이지** — 결과 목록에서 유저 클릭 시 기존 `/[username]` 프로필 페이지로 이동.
- **로그인 유저 전용** — 모든 로그인 유저가 사용 가능. 별도 프라이버시 설정 없음.

---

## 2. 사용자 여정

```
1. 로그인 유저가 홈(/) 진입
2. 스트라바 동기화 카드 아래에 검색창 + [검색] 버튼 노출
3. 아이디 또는 이메일 입력 → [검색] 클릭 (또는 Enter)
4. /search?q={검색어} 결과 페이지로 페이지 전환
5. 결과 목록: 아바타 + username + (지역·활동종목 등 부가정보)
6. 유저 카드 클릭 → /{username} 프로필 페이지로 이동
7. 결과 없으면 "검색 결과가 없어요" 빈 상태 표시
```

---

## 3. 기능 요구사항

### 3-1. 홈 검색 UI
| 항목 | 내용 |
|------|------|
| 위치 | 홈 페이지, 스트라바 동기화 카드 바로 아래 |
| 구성 | 텍스트 입력 + 검색 버튼 |
| 동작 | 제출 시 `/search?q={인코딩된 검색어}` 로 라우팅 (페이지 전환 방식) |
| 빈 검색어 | 제출 무시 (버튼 비활성 또는 no-op) |
| placeholder | "아이디 또는 이메일로 유저 검색" |

### 3-2. 검색 로직
| 항목 | 내용 |
|------|------|
| 검색 대상 | `users.username`, `users.email` |
| 매칭 | 대소문자 무시(ilike) 부분 일치. 검색어 전처리: trim + lowercase |
| 제외 | 본인 계정도 결과에 포함 (제외하지 않음) |
| username NULL 유저 | 이메일 매칭으로는 검색되지만, 프로필 URL이 없으므로 **결과에서 제외** (username 있는 유저만 노출) |
| 정렬 | 정확 일치 우선 → username 오름차순 |
| 최대 결과 수 | 30건 |

### 3-3. 검색 결과 페이지 (`/search`)
| 항목 | 내용 |
|------|------|
| 접근 | 로그인 필수 (비로그인 → 로그인 페이지 리다이렉트) |
| 표시 정보 | 아바타, username, 지역(region), 활동 종목(activity_types) |
| 클릭 | `/{username}` 프로필 페이지로 이동 |
| 빈 결과 | 빈 상태 문구 + 검색창 재노출 |
| q 파라미터 없음 | 검색창만 있는 빈 페이지 |

### 3-4. 보안·프라이버시
- 로그인 유저만 검색 API 호출 가능 (세션 검증).
- 검색 결과에 **이메일 원문을 노출하지 않는다** — 이메일로 검색은 가능하되, 결과 카드에는 username·아바타 등 공개 정보만 표시.
- 검색어는 서버에서 파라미터 바인딩으로 처리 (ilike 패턴 이스케이프: `%`, `_` 문자 제거 또는 이스케이프).
