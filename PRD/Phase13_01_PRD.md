# JAM! Phase 13 PRD — 미션 참가 확정 + 미션 상황(랭킹/현황) 화면

> 작성일: 2026-07-23
> 로직 원천: `jam-web/src/lib/missions/checker.ts`, `jam-web/src/app/api/missions/[id]/join/route.ts`
> 데이터 원천: `MissionRow`/`MissionCondition`(`src/types/database.ts`)

---

## 1. 배경 및 목표

### 현행 분석 (2026-07-23 코드 확인)

**이미 있는 것:**
- 참가/참가취소 API(`POST`/`DELETE /api/missions/[id]/join`), 참가·완료 테이블(`user_mission_participations`/`user_mission_completions`)
- 미션 4종 타입: `distance`(누적 거리) · `activity_count`(활동 횟수) · `poi_visit`(POI 방문) · `item_collect`(배지 수집)
- 진행률 계산(`checker.ts`)은 **`distance`·`activity_count`만 구현**되어 있고 `poi_visit`·`item_collect`는 항상 0 반환(미구현)
- 미션 상세 화면에 참가/취소 버튼 + "나의 진행 상황" 프로그레스 바 (본인 것만)

**발견한 버그 (이번 Phase에서 함께 수정):**
- `checker.ts`의 완료 판정(`achieved`)과 `user_mission_completions` INSERT가 **참가 여부(`user_mission_participations`)를 확인하지 않는다.** 참가 버튼을 누르지 않아도, 우연히 조건을 만족하는 활동을 하면 자동으로 미션이 완료 처리되고 보상까지 지급됨 — "참가해야 추적된다"는 이번 기획의 전제와 정면으로 어긋남.

### Phase 13 목표

1. **참가는 명시적 의사표시로만** — 목록에서 참가 버튼을 눌러야 진행상황 추적·완료·보상 지급 대상이 된다 (버그 수정).
2. **참가 취소 불가** — 한번 참가하면 되돌릴 수 없다 (기존 취소 API/버튼 제거).
3. **참가자 본인 진행상황 표시** — 기존에 있던 기능, 4종 미션 타입 전체로 확장(현재 `poi_visit`/`item_collect` 진행률 미구현 → 이번에 구현).
4. **"미션 상황" 화면 신설** — 참가한 미션의 상세 화면에 메뉴 추가. 참가자 전원(또는 상위 N명)의 달성 현황 + "내 순위/상태"를 보여준다.
5. **표시 방식은 미션마다 다를 수 있음** — 랭킹형(등수) / 숫자형(값) / 달성형(완료 여부) 세 가지를 지원.

## 2. 확정된 결정사항 (인터뷰 결과)

| 항목 | 결정 |
|------|------|
| 표시 방식 결정 주체 | **관리자가 미션 등록 시 직접 선택** (랭킹형/달성형 중) — "미션 성격에 따라 자동 판단"은 유저 스스로도 "다양한 미션을 시뮬레이션해보고 결정해야" 한다고 판단, 지금은 자동판단 대신 관리자 수동 지정으로 Phase 1을 확정하고 더 다양한 미션 유형(기간 내 POI 방문 횟수, 배지 발급 순서 등)은 Phase 2로 미룬다. |
| 공개 범위 | **관리자가 미션마다 "상위 N명" 지정** + **참가자 본인의 순위/상태는 N명 안에 없어도 항상 표시** |
| 완료자의 랭킹 내 표시 | Phase 1: 랭킹형은 **달성 순서대로 최상단 고정** (마라톤 완주 순서 방식). 유형별 세분화는 Phase 2. |
| 참가 필수화 버그 | **이번 Phase에서 함께 수정** — 참가자만 진행상황 추적·완료 처리 |

## 3. 사용자 여정

### 3-1. 참가

```
1. 유저가 미션 목록(/missions)에서 미션 선택
2. 미션 상세(/missions/[id]) 진입 — 달성 조건·보상 확인
3. [미션 참가하기] 버튼 클릭 → 참가 확정 (취소 불가 안내 문구 노출)
4. 이후 Strava 싱크마다 진행상황이 갱신됨 (참가자만)
```

### 3-2. 진행상황 확인 (본인)

```
1. 참가 중인 미션 상세 화면에서 "나의 진행 상황" 카드 확인
   - 랭킹형/숫자형 미션: 진행 수치 + 프로그레스 바
   - 달성형 미션(poi_visit/item_collect): 완료 여부 배지(달성/미달성)
```

### 3-3. 미션 상황(전체 현황) 확인

```
1. 참가 중인 미션 상세 화면에 "미션 상황" 메뉴 노출 (미참가자에게는 노출 안 함)
2. 탭하면 화면 전환 — 관리자가 지정한 상위 N명 리스트 + 내 순위/상태 (N명 밖이면 하단에 별도 고정 표시)
3. 랭킹형: 1위부터 순위·닉네임·진행값 나열. 달성한 유저는 달성 순서대로 최상단.
4. 달성형: 유저별 달성/미달성 상태만 나열 (순위 개념 없음)
```

## 4. 기능 요구사항

### 4-1. 참가 필수화 + 취소 불가

| 항목 | 내용 |
|------|------|
| 완료 판정 게이트 | `checker.ts`에서 `participationSet.has(mission.id)`일 때만 `achieved` 판정·완료 INSERT·보상 지급 진행 |
| 참가 취소 | `DELETE /api/missions/[id]/join` 제거, `MissionDetailClient`의 "참가 취소" 버튼 제거 |
| 참가 확정 UX | 참가 버튼 클릭 시 "참가 후 취소할 수 없어요" 확인 문구(간단한 confirm 또는 버튼 텍스트에 명시) |

### 4-2. 미션 상황 표시 방식 (관리자 설정)

| 필드 | 설명 |
|------|------|
| `status_display_type` | `'ranking'`(랭킹형) \| `'achievement'`(달성형) — 관리자가 미션 등록/수정 시 선택 |
| `visible_rank_count` | 상위 몇 명까지 공개할지 (관리자 설정, 비우면 전체 공개) |

- **랭킹형**: `distance`/`activity_count`처럼 누적 수치가 있는 미션에 적합. 진행값 내림차순 정렬, 달성자는 달성 순서대로 최상단.
- **달성형**: `poi_visit`/`item_collect`처럼 "했다/안 했다"만 있는 미션에 적합. 순위 없이 달성/미달성만 표시.
- 관리자가 미션 성격과 무관하게 선택해도 시스템은 막지 않는다(유연성 우선) — 다만 어드민 폼에서 미션 타입별 추천값을 기본 선택지로 보여준다.

### 4-3. poi_visit / item_collect 진행률 구현

- `poi_visit`: 유저가 해당 POI를 방문(POI 매칭 배지 발급 이력 또는 GPS 매칭 기록)했는지로 달성 여부 판단.
- `item_collect`: 유저가 해당 `badge_id`를 보유(`user_activity_badges` 또는 `inventory_items`)하는지로 달성 여부 판단.
- 두 타입 모두 진행값은 0/1(달성형)로 취급 — 진행률 프로그레스 바 대신 "달성/미달성" 배지로 표시.

### 4-4. 미션 상황 API/화면

- 신규 API: `GET /api/missions/[id]/status` — 참가자만 호출 가능(미참가자는 403). 응답: `{ type: 'ranking'|'achievement', entries: [...상위 N명...], me: {...} }`
- 신규 화면: 미션 상세에서 진입하는 별도 화면(`/missions/[id]/status` 또는 모달) — 랭킹형/달성형에 따라 다른 레이아웃.

## 5. 비범위 (Out of Scope — Phase 2 후보)

- 표시 방식 자동 판단(미션 조건 기반 추론)
- 새로운 복합 미션 타입 (기간 내 POI 방문 횟수, 배지 발급 순서/타이밍 경쟁 등)
- 완료자 랭킹 표시 방식의 미션 유형별 세분화 (Phase 1은 "달성 순서 최상단 고정"으로 통일)
- 실시간 갱신(웹소켓 등) — Phase 1은 페이지 새로고침/재방문 시 최신화
