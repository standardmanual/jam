## 7. 결론 — 미션 상황 페이지 타입 재검토

20개 샘플을 쭉 훑어본 결과, 표시 타입은 실질적으로 **두 갈래**로 수렴합니다.

### 관찰 1: "숫자형"과 "랭킹형"은 사실 같은 데이터, 다른 강조점일 뿐

`#1(거리)`, `#6(고도)`, `#8(새벽러닝 횟수)`, `#11(주말라이딩)`처럼 누적 수치가 있는 미션은 전부 **"참가자별 숫자 하나를 내림차순으로 나열한 목록"**이라는 동일한 화면 구조를 씁니다. 차이는:
- 화면에 **"1위 🥇" 같은 순위 배지를 강조**하느냐 (경쟁 프레이밍) → 랭킹형
- 아니면 그냥 **"내 진행값 vs 남들 진행값"만 담백하게 보여주느냐** (진행 확인 프레이밍) → 숫자형

→ **데이터/API/정렬 로직은 완전히 동일**하고, UI에서 순위 배지·강조색을 켜고 끄는 **표시 옵션 하나(`emphasize_rank: boolean`)** 로 충분합니다. 별도 타입으로 나눌 필요 없음.

### 관찰 2: "달성형"은 근본적으로 다른 데이터 형태

`#3(POI방문)`, `#4(배지수집)`, `#9(폭염완주)`, `#14(복합AND)`처럼 **진행값이 0/1(했다/안했다)뿐인 미션**은 정렬해도 의미가 없습니다 — "누가 먼저 했는지"(달성 순서)만 의미가 있고, 나머지는 전부 "미달성"으로 동률입니다. 이건 랭킹형과 **다른 화면**(달성자 명단 + 완료 시각, 미달성자는 순서 없이 별도 그룹)이 필요합니다.

### 최종 제안: 표시 타입은 2종으로 확정

| 타입 | 대상 | 화면 |
|------|------|------|
| **`ranking`** (숫자·랭킹 통합) | 누적/최고값이 있는 모든 미션 (§1의 #1,2 / §2의 #6,7,8 / §3 전부 / §4의 #15,17 / §6) | 참가자를 값 기준 정렬 + `emphasize_rank` 옵션으로 순위 강조 여부만 조절 |
| **`achievement`** | 0/1 판정 미션 (§1의 #3,4 / §2의 #5,9,10 / §3의 #13,14 / §4의 #16) | 달성자(달성 시각순) + 미달성자(순서 없음) 두 그룹 |

→ **Phase13_01_PRD.md §4-2의 `status_display_type: 'ranking' | 'achievement'` 결정을 그대로 유지**하되, `ranking` 타입에 `emphasize_rank`(또는 동일한 의미의 UI 토글) 옵션을 추가하는 걸 권장합니다. 애초 인터뷰에서 나온 "3가지 케이스(등수랭킹/숫자값/달성값)"는 결국 **"랭킹형 안의 표현 옵션 2가지 + 별개인 달성형 1가지"**로 정리되는 셈입니다.

### 남는 과제 (신규 조건 타입 — Phase 2)

- `#15` 순수 랭킹(목표치 없이 기간 종료 시 등수 자체가 보상 기준)은 현재 미션 데이터 모델(`condition_json`에 목표치가 필수)과 안 맞음 — "목표치 없는 미션" 개념 추가 필요.
- `#17` 기간 내 POI 방문 **횟수** 카운트는 현재 `poi_visit` 로직(방문 여부만)에 없음 — 신규 조건.
- `#18/19` 팀 미션은 `user_mission_participations`를 개인 단위에서 팀 단위로 확장해야 함 — 스키마 변경 필요, Phase 2.
- `#20` 포인트 연동은 `mission_type`에 `'points'` 추가 + 포인트 거래 내역 기간 집계 로직 필요.
