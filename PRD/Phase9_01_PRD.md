# JAM! Phase 9 PRD — 구글 로그인 온보딩: 아이디 생성 + 프로필 이미지

> 작성일: 2026-07-14  
> 범위: 첫 로그인 시 아이디(username) 생성 온보딩, 구글 프로필 이미지 자동 적용, 프로필 페이지 편집 기능

---

## 1. 배경 및 목표

현재 구글 OAuth 로그인 후 `display_name`(구글 이름)이 자동으로 저장되고, 온보딩 페이지는 활동 유형 선택만 존재한다. 유저를 식별하는 고유 아이디가 없어 추후 팔로우, 트레이드, 소셜 기능 구현이 불가능하다.

Phase 9의 목표:
- **아이디(username) 도입** — 유저를 식별하는 고유 핸들. `display_name`을 대체.
- **첫 로그인 시 필수 아이디 생성** — 온보딩 화면에서 아이디를 만들어야만 서비스 진입 가능.
- **구글 프로필 이미지 자동 적용** — 첫 로그인 시 구글 계정 사진이 `avatar_url`로 저장됨.
- **프로필 페이지에서 수정 가능** — 아이디(username) 변경 + 프로필 사진 교체(파일 업로드).

## 2. 사용자 여정

### 2-1. 첫 로그인 (신규 유저)

```
1. 유저가 로그인 페이지에서 [구글로 계속하기] 버튼 클릭
2. 구글 OAuth 동의 화면 → 승인
3. /auth/callback 로 리다이렉트
4. Supabase가 구글 프로필 정보(이름, 이메일, 사진 URL) 수신
5. users 테이블에 신규 행 생성:
   - avatar_url = 구글 프로필 사진 URL
   - username = NULL (아직 미설정)
6. username이 NULL이므로 → /onboarding 으로 리다이렉트
7. 온보딩 화면: 아이디 입력 + 실시간 유효성 검사 + 사용 가능 여부 표시
8. [생성하기] 클릭 → users.username 저장 → / (홈) 으로 이동
```

### 2-2. 재로그인 (기존 유저 — username 있음)

```
1. /auth/callback 에서 username 존재 확인
2. username이 NOT NULL이면 → / (홈) 으로 바로 이동
3. 온보딩 화면 노출하지 않음
```

### 2-3. 프로필 페이지 편집 (이미 로그인한 유저)

```
1. 유저가 /profile 에서 [편집] 버튼 탭
2. 편집 화면:
   a. 현재 프로필 사진 표시 + [사진 변경] 버튼
   b. 현재 아이디 표시 + 수정 가능 입력 필드
3. 아이디 변경 시: 실시간 유효성 + 중복 체크
4. 사진 변경 시: 갤러리/카메라에서 선택 → Supabase Storage 업로드 → avatar_url 갱신
5. [저장] 클릭 → 홈 또는 프로필로 이동
```

## 3. 아이디(username) 규칙

인스타그램 아이디 네이밍 규칙을 준용한다.

| 규칙 | 상세 |
|------|------|
| 허용 문자 | 영문 소문자(a-z), 숫자(0-9), 점(.), 언더스코어(_) |
| 길이 | 1자 이상 30자 이하 |
| @ 금지 | `@` 기호 사용 불가 |
| 연속 점 금지 | `..` 처럼 점이 연속으로 올 수 없음 |
| 시작/끝 점 금지 | 점으로 시작하거나 끝나면 안 됨 |
| 대소문자 | 입력은 자유이나 저장은 소문자로 통일 (case-insensitive unique) |
| 공백 금지 | 공백 문자 불가 |

### 3-1. 유효성 에러 메시지

| 상황 | 메시지 |
|------|--------|
| 비어있음 | "아이디를 입력해 주세요" |
| 30자 초과 | "30자 이하로 입력해 주세요" |
| 허용되지 않는 문자 | "영문, 숫자, ., _ 만 사용할 수 있어요" |
| 점으로 시작/끝 | "점(.)으로 시작하거나 끝날 수 없어요" |
| 연속된 점 | "점(.)을 연속으로 사용할 수 없어요" |
| 이미 사용 중 | "이미 사용 중인 아이디예요" |
| 사용 가능 | "사용 가능한 아이디예요 ✓" (초록색) |

## 4. 온보딩 화면 UI 명세

### 4-1. 화면 구성

```
┌─────────────────────────────┐
│  [헤더 영역 없음]             │
│                             │
│  ╔═══════════════════════╗  │
│  ║   👤 (프로필 이미지)    ║  │
│  ║  (구글 사진 또는 기본)   ║  │
│  ╚═══════════════════════╝  │
│                             │
│  JAM! 아이디를 만들어 주세요   │
│  (서브타이틀)                │
│                             │
│  ┌───────────────────────┐  │
│  │  @  [아이디 입력]     │  │
│  └───────────────────────┘  │
│  [유효성 메시지]              │
│                             │
│  ┌───────────────────────┐  │
│  │      생성하기          │  │
│  └───────────────────────┘  │
│                             │
└─────────────────────────────┘
```

### 4-2. 인터랙션

- 입력 필드에 `@` 접두사 고정 표시 (입력값에는 포함하지 않음)
- 입력할 때마다(debounce 500ms) 유효성 검사 + 중복 확인 API 호출
- 유효 + 사용 가능: 입력 필드 테두리 초록, "사용 가능" 메시지
- 오류: 입력 필드 테두리 빨강, 에러 메시지
- 중복 확인 중: 로딩 스피너
- [생성하기] 버튼: 유효하지 않거나 중복 시 비활성화(disabled)

## 5. API 명세

### 5-1. 아이디 중복 확인 (GET)

```
GET /api/username/check?username={value}
```

**응답:**
```json
{ "available": true }
{ "available": false }
```

- 인증 불필요 (온보딩 전이므로)
- 포맷 유효성 검사 서버에서도 재검증 (클라이언트 우회 방지)
- `username`을 소문자로 변환 후 조회

### 5-2. 아이디 저장 (POST)

```
POST /api/onboarding/complete
Body: { "username": "sihyun.runs" }
```

**응답:**
```json
{ "success": true }
{ "error": "DUPLICATE" | "INVALID_FORMAT" | "UNAUTHORIZED" }
```

- 인증 필요 (Supabase JWT)
- 트랜잭션: `users.username` 업데이트 + `users.display_name` 삭제(컬럼 제거됨)

### 5-3. 프로필 이미지 업로드 (POST)

```
POST /api/profile/avatar
Content-Type: multipart/form-data
Body: { "file": [이미지 파일] }
```

**응답:**
```json
{ "avatar_url": "https://...supabase.co/storage/..." }
```

- 허용 파일 형식: JPEG, PNG, WebP
- 최대 크기: 5MB
- Supabase Storage `avatars` 버킷에 저장
- 경로: `{user_id}/{timestamp}.{ext}`
- 기존 파일 자동 삭제 (덮어쓰기)

## 6. 프로필 페이지 편집 기능

현재 `/profile` 페이지에 편집 기능을 추가한다.

### 6-1. 편집 화면 진입

- [편집] 버튼(오른쪽 상단) 탭 시 편집 모드로 전환
- 별도 페이지 이동 없이 인라인 편집 또는 `/profile/edit` 서브 페이지 (구현 편의에 따라 결정)

### 6-2. 편집 가능 항목

| 항목 | 설명 |
|------|------|
| 프로필 사진 | 현재 사진 탭 시 갤러리 선택 → 업로드 |
| 아이디(username) | 실시간 유효성/중복 체크 포함 |

### 6-3. 저장 동작

- 변경사항 없으면 [저장] 비활성화
- 저장 완료 후 프로필 페이지로 복귀
- 변경 중 뒤로가기 시 "변경사항을 저장하지 않겠습니까?" 경고 (선택 사항 — MVP에서 생략 가능)

---

## 7. 기존 데이터 처리 (마이그레이션)

- 기존 유저들(`username = NULL`)은 다음 로그인 시 온보딩 화면을 다시 보게 됨
- `display_name` 컬럼은 `username`으로 rename. 기존 데이터는 마이그레이션 SQL에서 처리.
- 구글 프로필 이름이 username 규칙에 맞지 않을 수 있으므로 자동 변환 금지 — 반드시 유저가 직접 입력하게 함
