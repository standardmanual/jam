## [수정] Strava 싱크 파이프라인 (`src/lib/strava/sync.ts`)

### 버그 원인 사슬 (초기화 후 첫 싱크에서 발견)

`last_synced_at=NULL` 백필 → 전체 이력 활동마다 Streams API 호출 + 드랍(활동당 최소 1개) → 요청 타임아웃(무한대기) → 드랍은 서버에서 계속 삽입(아이템 폭주) → 응답 없음으로 재시도 → 커서 미갱신 상태라 같은 활동 재처리(피드 중복) → 배지 엔진은 드랍 루프 이후 단계라 실행 전 중단(액티비티배지 미발급).

### 수정 4종

| 수정 | 내용 |
|------|------|
| **동시 싱크 잠금** | 처리 시작 전에 `last_synced_at`을 낙관적 잠금으로 선점 갱신. 같은 커서의 두 번째 요청은 갱신 0행 → 즉시 종료. 종료 시 재갱신하지 않음(처리 중 업로드 활동 누락 갭 방지) |
| **첫 싱크 드랍 1회** | `initial_sync_done=false`면 최신 활동 1건만 드랍 — "10초 첫 보상" 스펙. 과거 이력 전체에 드랍 폭주 차단 |
| **드랍 활동 상한** | 일반 싱크도 최신 활동 최대 3건만 드랍 시도 (`MAX_DROP_ACTIVITIES_PER_SYNC=3`) |
| **POI 매칭 상한** | Streams API 호출을 최신 활동 10건으로 제한 (`MAX_POI_MATCH_ACTIVITIES_PER_SYNC=10`) — Strava rate limit·타임아웃 방지 |

### 운영 유의사항

- 잠금이 커서를 선점하므로 처리 중 크래시가 나면 해당 배치 활동은 배지·드랍 평가를 건너뛴다 (트레이드오프 — 중복 발급보다 안전).
- 첫 싱크의 액티비티배지 평가는 배치 전체를 그대로 사용 (드랍만 1회 제한) — Common 게이트·소급 발급은 기존과 동일.
- 오염된 테스트 계정(중복 드랍·피드)은 어드민 유저 초기화를 다시 실행하면 정리된다 (초기화가 피드·아이템·드랍상태 전부 삭제).
