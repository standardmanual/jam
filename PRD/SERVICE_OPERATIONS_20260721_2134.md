## [수정] Strava 싱크 무한대기 (진짜 근본 원인)

### 재진단

이전 버전(`34e10f3`)에서 동시 싱크 잠금·드랍/POI 개수 상한을 적용했으나, 배포 후 재테스트에서도 동일한 "무한대기" 증상이 재현됨. 추가 조사 결과 진짜 원인은 별개였다:

1. **`/api/strava/sync` 라우트에 `maxDuration` 미설정** — Vercel 기본 타임아웃(플랜별 10~15초) 적용. 백필 싱크는 POI 매칭(외부 Strava Streams API 순차 호출) + 드랍/배지 평가가 겹쳐 수 초~수십 초가 걸릴 수 있어 함수가 도중에 강제 종료됨.
2. **Strava fetch 요청에 타임아웃 없음** — Strava 응답이 지연되면 `fetch()`가 무기한 대기. 10개 활동을 **순차** 조회하는 구조라 지연이 누적되면 더욱 악화.
3. 함수가 강제 종료되기 전까지 이미 실행된 **드랍/DB 쓰기는 그대로 커밋**됨 — 이것이 "무한대기 + 아이템배지 다량 발급 + 피드 중복"이 함께 나타난 이유. 강제 종료로 응답을 못 받은 클라이언트는 "무한대기"로, 그 사이 부분 실행된 로직은 DB에 이미 반영됨.
4. 액티비티배지는 파이프라인 후반(9단계)이라 함수가 그 전에 종료되면 **전혀 발급되지 않음** — 별도 버그가 아니라 같은 타임아웃의 결과.

### 수정 3종

| 수정 | 내용 |
|------|------|
| `maxDuration` 명시 | `/api/strava/sync`(수동) 60초, `/api/cron/sync`(전체 배치) 300초 — Vercel 기본 타임아웃보다 넉넉히 확보 |
| **POI Streams 병렬 조회** | 최신 10건의 Strava Streams 호출을 `Promise.all`로 병렬화 (기존: 순차 for-loop). 배지 발급(DB 쓰기)은 순서 보장을 위해 순차 유지 |
| **Strava fetch 타임아웃 8초** | `stravaFetch`·`getActivityStreams`·`refreshStravaToken` 전체에 `AbortSignal.timeout(8000)` 적용 — Strava 응답 지연 시 무기한 대기 대신 8초 후 실패 처리(개별 활동은 건너뛰고 계속 진행) |
